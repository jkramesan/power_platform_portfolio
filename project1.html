Customer Tracking Project

Why We Created This Solution

The Customer Service Ticketing System was developed to streamline customer support processes by providing an efficient way to track and manage service requests. It enables better response times, transparency, and customer satisfaction by centralizing customer interactions.

Purpose of the Project

  • To provide a structured system for managing customer support tickets.
  • To enhance efficiency in handling customer issues.
  • To enable better tracking of ticket status and resolution times.
  • To offer an integrated system using Power Platform’s model-driven app capabilities.

Tables & Relationships Explanation

1. Customers Table (Primary Table)

Purpose: Stores customer details who raise service tickets.

Column Name Data Type Description
Customer ID (Primary Key) Autonumber Unique identifier for each customer
Full Name Text Customer's full name
Email Email Customer's email address
Phone Phone Contact number of the customer
Address Text Customer's address (Optional)
Created On Date/Time Auto-generated when a new customer is added

Relationships: Customers (1) → (M) Tickets: A single customer can create multiple support tickets.

2. Tickets Table (Core Table)

Purpose: Tracks customer issues and service requests.

Column Name Data Type Description
Ticket ID (Primary Key) Autonumber Unique identifier for each ticket
Title Text Short summary of the issue
Description Multiline Text Detailed description of the problem
Resolution Notes Multiline Text Notes on how the issue was resolved

Relationships: Tickets (M) → (1) Customers, Tickets (M) → (1) Agents, Tickets (M) → (1) Status.

3. Agents Table (Support Team Members)

Purpose: Stores details of support agents handling customer tickets.

Column Name Data Type Description
Agent ID (Primary Key) Autonumber Unique identifier for each agent
Full Name Text Agent’s full name
Email Email Official email of the agent
Department Choice (IT, Billing, Technical) Specifies the team the agent belongs to
Availability Status Choice (Available, Busy, Offline) Indicates agent’s work status
Created On Date/Time Timestamp when the agent was added

4. Status Table (Lookup Table)

Purpose: Stores possible states for service tickets.

Column Name Data Type Description
Status ID (Primary Key) Autonumber Unique identifier for each status
Status Name Choice Options: New, In Progress, Resolved, Closed

Key Features & Implementation

1. Subgrid Implementation

Where? On the Customer Form, showing all tickets linked to a customer.

Why? This helps support teams quickly access and manage customer issues.

Steps:

  • Open Power Apps > Dataverse > Tables > Customers
  • Go to Forms and open the Main Form
  • Add a Subgrid Component and link it to the Tickets Table
  • Set the View to display Active Tickets
  • Save & Publish

2. Power Fx / Business Rule Formulas

Functionality Formula / Rule
Auto-Assign Ticket to an Agent If Agent.Availability = "Available", assign ticket to that agent
Set Ticket Creation Date Automatically Ticket.CreatedOn = Now()
Auto-Update Ticket Status If Ticket.ResolutionNotes is filled, change Status to "Resolved"
Calculate Resolution Time Ticket.ResolutionTime = Ticket.ResolvedOn - Ticket.CreatedOn

Scenario: How This Project is Used

A retail company needs to track customer complaints and ensure timely resolution. The workflow is as follows:

  • Customers submit issues via email or a self-service portal.
  • The support team logs the issue in the Tickets Table.
  • The system auto-assigns available agents.
  • Agents update ticket status, and customers receive notifications.
  • The manager reviews reports on resolution times for performance tracking.
